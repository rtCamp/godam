# GoDAM Plugin — Unit Testing Setup Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Establish a complete unit-testing stack (PHP + JS) for the GoDAM WordPress plugin that lives in git but is excluded from the WordPress.org build artifact, with CI gating and progressive coverage of plugin subsystems.

**Architecture:** Two parallel test runners (PHPUnit for server-side PHP, Jest for the React/block JS layer) wired through Composer + `@wordpress/scripts` (already devDeps). Tests run locally via `wp-env`, in CI via a new GitHub Actions workflow, and are stripped from the release zip via `.distignore` + `.gitattributes`. Coverage is generated by both runners and uploaded to Codecov for tracking.

**Tech Stack:** PHPUnit 9.6, wp-phpunit 6.2, Yoast PHPUnit-Polyfills, Brain Monkey + Mockery (pure-unit isolation), WP-CLI scaffold (`wp scaffold plugin-tests`), Jest 29 (via `@wordpress/jest-preset-default`), React Testing Library, `@wordpress/env` for the WP integration runtime, Codecov for reporting, GitHub Actions for CI.

## 1. Tech Stack

Almost everything required is already in `composer.json` and `package.json` — this plan formalises usage rather than introducing a new ecosystem.

### 1.1 PHP — Server-Side Tests

| Concern | Tool | Already installed? | Notes |
| --- | --- | --- | --- |
| Test framework | `phpunit/phpunit ^9.6` | Yes (composer.json:8) | Stay on 9.6 — 10/11 drops PHP 7 and breaks wp-phpunit compatibility |
| WP test harness | `wp-phpunit/wp-phpunit ^6.2` | Yes (composer.json:9) | Provides `WP_UnitTestCase`, factories, transactional DB |
| PHP version compat | `yoast/phpunit-polyfills ^1.0` | Yes (composer.json:10) | Required by wp-phpunit on PHP 7.4–8.4 |
| Pure-unit mocking | `brain/monkey ^2.6` + `mockery/mockery ^1.6` | **No — add** | Lets us mock WP functions without booting WordPress; needed for "fast" unit tests |
| Static analysis (out of scope here but adjacent) | `szepeviktor/phpstan-wordpress` | No — see future plan | Tracked in [GoDAM tech-debt backlog](https://github.com/rtCamp/godam/issues) |
| Coverage driver | Xdebug or PCOV | Provided by `wp-env`/CI runner | Prefer PCOV in CI (10× faster) |
| Scaffolding | `wp-cli/scaffold-command` | Bundled with wp-cli | Run once to seed `tests/bootstrap.php` and `tests/phpunit.xml.dist` |

**Two distinct PHP test tiers** (this is the most common pitfall — keep them separate):
- **`tests/phpunit/unit/`** — Pure unit tests. No WordPress loaded. Use Brain Monkey to fake `add_action`, `apply_filters`, `get_option`, etc. Fast (sub-second per file). Ideal for helpers, traits, REST controllers' validation logic, schema builders, enum classes.
- **`tests/phpunit/integration/`** — Boot WordPress via `wp-phpunit`. Use `WP_UnitTestCase` and factories. Slower (multi-second startup). Needed when code hits `$wpdb`, post/user objects, cron, options, hooks-with-side-effects.

Both tiers share a single composer entry but have separate `phpunit.xml.dist` configs (`phpunit-unit.xml.dist`, `phpunit-integration.xml.dist`) and separate npm/composer scripts.

### 1.2 JavaScript — Block & Admin UI Tests

| Concern | Tool | Already installed? | Notes |
| --- | --- | --- | --- |
| Test framework | Jest (via `@wordpress/jest-preset-default ^12.28`) | Yes (package.json:79) | `wp-scripts test-unit-js` is the canonical runner |
| React testing | `@testing-library/react ^14` + `@testing-library/jest-dom ^6` | **No — add** | Industry standard; integrates with Jest preset |
| User interaction | `@testing-library/user-event ^14` | **No — add** | Needed for click/type simulation in block edit tests |
| DOM env | `jest-environment-jsdom` | Bundled by preset | No action |
| Coverage | Jest built-in (`--coverage`) | Yes | Outputs `lcov` consumable by Codecov |

JS tests live alongside source: `assets/src/blocks/<block-name>/__tests__/edit.test.js`. The Jest preset already understands this layout — no extra config needed beyond a project-level `jest.config.js` to point `testPathIgnorePatterns` at `node_modules`, `vendor`, and `assets/build`.

### 1.3 Runtime / Orchestration

- **Local PHP integration runtime:** `@wordpress/env` (already installed, package.json:75). Provides a Dockerised WP + MySQL on demand. The integration phpunit config will point `WP_TESTS_DIR` at the wp-env container's WP install.
- **Local PHP unit runtime:** none — Brain Monkey-based tests run on the host PHP CLI.
- **CI:** GitHub Actions, new workflow `tests_on_pull_request.yml` running PHP matrix (7.4, 8.1, 8.4) × WP matrix (6.5, latest), plus a single Jest job.
- **Coverage upload:** Codecov action (`codecov/codecov-action@v5`) — public projects free, no token needed for OSS, encrypted token for private. GoDAM is public so this works out of the box.

### 1.4 What we deliberately leave out (YAGNI)

- **End-to-end / Playwright** — separate plan; not unit testing. `@wordpress/scripts test-e2e` is available when we get there.
- **Mutation testing (Infection)** — premature until baseline coverage exists.
- **Codeception / WP Browser** — adds a parallel BDD framework on top of PHPUnit, not worth the duplication for this codebase.
- **PHPStan / Psalm config changes** — `wp-phpstan` skill applies; tracked separately.

## 2. Code Coverage Sections

Coverage is rolled out in priority order. Each section ships independently — do **not** wait for "all PHP done" before starting JS. The order below reflects highest-risk-per-LOC, not directory layout.

### Section A — Pure-Logic PHP (Tier 1, no WordPress)

Fastest to write, fastest to run, highest leverage. Target ≥ 90% line coverage for this section.

| Area | Files | Test type | Owner files (illustrative) |
| --- | --- | --- | --- |
| Enums | `inc/classes/enums/*.php` | unit | `tests/phpunit/unit/Enums/` |
| Traits | `inc/classes/traits/*` and `inc/traits/*` | unit (one consumer fixture per trait) | `tests/phpunit/unit/Traits/` |
| Helpers | `inc/helpers/*.php` | unit, Brain Monkey for WP fns | `tests/phpunit/unit/Helpers/` |
| Validation / schema in REST controllers | `inc/classes/rest-api/class-*.php` — `get_*_schema()`, `validate_*`, `permission_callback` | unit | `tests/phpunit/unit/RestApi/` |
| Migration version comparators | `inc/classes/migrations/` | unit | `tests/phpunit/unit/Migrations/` |

### Section B — REST API Endpoint Behaviour (Tier 2, WP loaded)

REST routes are user-facing contracts. Break these and you break the React admin. Target ≥ 80% per route.

- One test class per REST controller in `inc/classes/rest-api/` (23 files identified).
- For each route: happy path, permission denial, malformed payload, success-with-side-effect (asserted via factory-created posts/options).
- Use `WP_REST_Request` directly, not HTTP. Assert response status, schema, and DB state.
- Particularly important: `class-transcoding.php`, `class-analytics.php`, `class-media-library.php`, `class-settings.php`, `class-engagement.php`.

### Section C — Form Integrations (Tier 2 with conditional skipping)

GoDAM ships hook-based integrations with 8+ form plugins (CF7, GF, NF, WPForms, Fluent Forms, Forminator, SureForms, Everest, Metform, Gravity Forms). Each integration only loads if the host plugin is active.

- One test class per integration in `tests/phpunit/integration/FormIntegrations/`.
- Skip via `markTestSkipped()` when the host plugin isn't installable in the test runtime — keeps CI green without requiring all form vendors.
- Where possible, mock the host plugin's public API surface rather than installing it. CF7 and WPForms have stable enough public functions to mock cleanly.
- Cover: hook registration on load, payload-to-GoDAM-media transformation, and the GoDAM media attachment side-effects.

### Section D — Blocks (Jest)

Block components are React. Target ≥ 70% per block — UI tends to have lower-value coverage thresholds.

- Test layout: `assets/src/blocks/<block-name>/__tests__/edit.test.js` and `save.test.js`.
- Cover: default attribute rendering, attribute mutation via `setAttributes`, conditional UI branches, and `save()` output stability (snapshot-based — use sparingly).
- Mock `@wordpress/block-editor` and `@wordpress/data` selectively; the Jest preset includes these mocks.
- Block-extensions in `assets/src/block-extensions/` get the same treatment.

### Section E — Admin / Transcoder PHP (Tier 2)

The transcoder is the most stateful part of the plugin. Highest defect cost.

- `admin/class-rtgodam-transcoder-handler.php` — covers the HTTP callback handlers, signature verification, attachment update flow. Use `WP_Http_*` mocks and `wp_remote_request` filter to intercept outbound calls.
- `admin/class-rtgodam-transcoder-admin.php` — bulk action handlers, settings registration.
- `admin/class-rtgodam-retranscodemedia.php` — re-transcode queue logic.
- `admin/class-rtgodam-transcoder-rest-routes.php` — overlaps with Section B.

### Section F — Cron / Background Jobs (Tier 2)

- `inc/classes/cron-jobs/*` — assert `wp_schedule_event` registration on activation, that the scheduled callback runs, and that the next-run gets re-scheduled.
- Action Scheduler integrations (per composer.json:26): assert actions are queued with correct hook/args.

### Section G — Plugin Bootstrap (Tier 2)

- `inc/classes/class-plugin.php`, `class-assets.php`, `class-rewrite.php`, `class-pages.php`, `class-deactivation.php`.
- Cover: singleton boot, asset registration handles + dependencies, rewrite rule flush trigger, deactivation cleanup.
- Don't aim for line coverage here — aim for "the plugin can boot under WP test runtime without warnings or fatals."

### Section H — Helpers, Templates, Embed Logic (Tier 1 or Tier 2 as needed)

- `inc/classes/class-embed.php`, `class-video-embed.php`, `class-video-preview.php`, `class-video-metadata.php`, `class-video-engagement.php`, `class-video-editor-form-layer-handler.php` — mostly Tier 2, factory-driven.
- `inc/templates/*` — these are rendered partials; coverage via integration tests of the classes that include them.

### Coverage targets summary

| Section | Tier | Target |
| --- | --- | --- |
| A — Pure logic | Unit | ≥ 90% |
| B — REST routes | Integration | ≥ 80% |
| C — Form integrations | Integration | ≥ 60% (gated on plugin availability) |
| D — Blocks (JS) | Jest | ≥ 70% |
| E — Admin/Transcoder | Integration | ≥ 75% |
| F — Cron/background | Integration | ≥ 70% |
| G — Bootstrap | Integration | smoke-only |
| H — Embed/video classes | Integration | ≥ 70% |

Plugin-wide target after section G+H land: **70% line / 60% branch (PHP)** and **65% line (JS)**. Codecov reports both numbers per PR.

## 3. Everything Else

### 3.1 Excluding tests from the build artifact

The release pipeline (`.github/workflows/release_on_tag.yml`) uses `10up/action-wordpress-plugin-deploy@2.3.0`, which respects `.distignore`. The push deploy (`deploy_on_push.yml`) uses `rtcamp/action-deploy-wordpress`, which **also** respects `.distignore`. Both already run `composer install --no-dev`, so `phpunit/phpunit` and friends are dropped from the production `vendor/` automatically.

**Required `.distignore` additions** (current file at .distignore:1-33):

```
# Tests
tests/
phpunit.xml.dist
phpunit-unit.xml.dist
phpunit-integration.xml.dist
jest.config.js
.phpunit.result.cache
coverage/
.codecov.yml
docs/
```

**Belt-and-braces — add `.gitattributes`** so anyone doing `git archive` instead of using the deploy actions also gets a clean tree:

```gitattributes
/tests              export-ignore
/docs               export-ignore
/phpunit*.xml.dist  export-ignore
/jest.config.js     export-ignore
/.codecov.yml       export-ignore
/.github            export-ignore
```

**Verification step:** after wiring tests, run `npm install -g @wp-cli/wp-cli` then `wp dist-archive .` locally and inspect the produced zip to confirm `tests/` is absent. Add this verification as a CI step (`Verify build artifact` job in `release_on_tag.yml`) so a regression in `.distignore` fails the release rather than silently shipping tests.

### 3.2 CI workflow

New file: `.github/workflows/tests_on_pull_request.yml`. Mirrors the existing `phpcs_on_pull_request.yml` convention.

Matrix:
- PHP: 7.4, 8.1, 8.4 (matches `phpcompatibility-wp` floor)
- WordPress: 6.5, latest (matches `wp-phpunit ^6.2` floor)
- Three jobs: `php-unit` (fast, no matrix), `php-integration` (matrixed), `js-unit` (single node version per `.nvmrc`)

Steps per job (integration job shown):
1. `actions/checkout@v6`
2. Set up PHP with PCOV via `shivammathur/setup-php@v2`
3. `composer install --prefer-dist --no-progress`
4. Start MySQL service container
5. `bash bin/install-wp-tests.sh wordpress_test root '' 127.0.0.1 ${{ matrix.wp }}`
6. `composer run test:integration -- --coverage-clover=coverage.xml`
7. `codecov/codecov-action@v5` with `flags: php-integration`

Failure on any matrix cell blocks merge once branch protection is configured (see 3.6).

"test:js:watch": "wp-scripts test-unit-js --watch",

```

```

Document in `CONTRIBUTING.md` (already exists at the repo root) under a new "Running tests" section with copy-pasteable commands and the wp-env startup sequence.

### 3.4 Bootstrap files (`tests/bootstrap.php` strategy)

Two bootstraps, mirroring the two tiers:

- `tests/phpunit/bootstrap-unit.php` — loads Brain Monkey (`Brain\Monkey\setUp()` / `tearDown()` in a base TestCase), autoloads classes from `inc/` via a lightweight `require_once` map (no Composer autoload needed for plugin classes since GoDAM doesn't use PSR-4). Provides a `GoDAM\Tests\Unit\TestCase` base class.
- `tests/phpunit/bootstrap-integration.php` — locates `WP_TESTS_DIR`, loads `wp-tests-config-sample.php`-derived config, requires plugin via `muplugins_loaded` filter, then loads `wp-phpunit/includes/bootstrap.php`.

Provide `bin/install-wp-tests.sh` (the canonical WP-CLI scaffold script). Generate by running `wp scaffold plugin-tests godam` once and then customising. Commit it; CI invokes it directly.

### 3.5 Coverage reporting

- `.codecov.yml` at repo root: define flags `php-unit`, `php-integration`, `js-unit`. Set carry-forward so jobs that skip on a PR don't tank the score.
- Coverage threshold gate: PR fails if total coverage drops more than 1 % vs. base. Soft-fail for the first 60 days while the suite stabilises (Codecov supports `informational: true`).
- Badge in `README.md` near the existing build badges.

### 3.6 Branch protection & required checks

Once the workflow is green for two weeks (give it time to flush flakes), update GitHub branch protection on `develop`:
- Required status checks: `php-unit`, `php-integration (php 8.1, wp latest)`, `js-unit`.
- Don't require every matrix cell — only the "modern" one — to keep the gate practical.

### 3.7 Pre-commit hook integration

The repo uses Husky (.husky/) and lint-staged (.lintstagedrc.js). Add:
- Run `npm run test:js -- --findRelatedTests <changed JS files>` on commit (fast — only related Jest tests).
- Do **not** run PHPUnit on commit — too slow. Run on `pre-push` instead, gated on a `tests` directory existing.

### 3.8 Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| Brain Monkey state leaks between tests | Enforce `setUp`/`tearDown` in base TestCase; CI runs `--process-isolation` for unit suite at first to detect leaks. |
| Integration suite slow → devs skip it | Keep Tier 1 fast (< 5 s total) and document `composer test:unit` as the "before you push" command. |
| Tests sneak into the zip | Verification job in `release_on_tag.yml` greps the artifact for `tests/`; fails the release if found. |
| Form-integration tests pin to a specific host-plugin version | Mock public API only; pin host-plugin versions in a comment in the test for traceability. |
| Coverage gate blocks valid PRs (typo fixes etc.) | Codecov's `informational: true` for the first 60 days; revisit after baseline. |

```yaml
name: Tests on Pull Request

on:
  pull_request:
    branches: [develop, main]
  workflow_dispatch:

permissions:
  contents: read

jobs:
  php-unit:
    name: PHP unit (${{ matrix.php }})
    runs-on: ubuntu-latest
    strategy:
      matrix:
        php: ['7.4', '8.1', '8.4']
    steps:
      - uses: actions/checkout@v6
      - uses: shivammathur/setup-php@v2
        with:
          php-version: ${{ matrix.php }}
          coverage: pcov
      - run: composer install --prefer-dist --no-progress
      - run: composer test:unit -- --coverage-clover=coverage-unit.xml
      - uses: codecov/codecov-action@v5
        with:
          files: coverage-unit.xml
          flags: php-unit-${{ matrix.php }}

  php-integration:
    name: PHP integration (php ${{ matrix.php }}, wp ${{ matrix.wp }})
    runs-on: ubuntu-latest
    services:
      mysql:
        image: mysql:8.0
        env:
          MYSQL_ROOT_PASSWORD: root
        ports:
          - 3306:3306
        options: --health-cmd="mysqladmin ping" --health-interval=10s --health-timeout=5s --health-retries=5
    strategy:
      matrix:
        include:
          - php: '7.4'
            wp: '6.5'
          - php: '8.1'
            wp: 'latest'
          - php: '8.4'
            wp: 'latest'
    steps:
      - uses: actions/checkout@v6
      - uses: shivammathur/setup-php@v2
        with:
          php-version: ${{ matrix.php }}
          coverage: pcov
          extensions: mysqli, zip
      - run: composer install --prefer-dist --no-progress
      - run: bash bin/install-wp-tests.sh wordpress_test root root 127.0.0.1 ${{ matrix.wp }}
      - run: composer test:integration -- --coverage-clover=coverage-int.xml
      - uses: codecov/codecov-action@v5
        with:
          files: coverage-int.xml
          flags: php-integration-${{ matrix.php }}-${{ matrix.wp }}

  js-unit:
    name: JS unit
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-node@v6
        with:
          node-version-file: '.nvmrc'
          cache: 'npm'
      - run: npm ci
      - run: npm run test:js -- --coverage
      - uses: codecov/codecov-action@v5
        with:
          files: coverage/js/lcov.info
          flags: js-unit
```
